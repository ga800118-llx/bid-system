# 用户管理页样板说明

## 目的

`/system/user` 当前承担基础平台后台列表页样板角色。

这份说明不是记录“用户管理功能点”，而是明确：

- 哪些结构和组件已经在该页验证通过
- 后续哪些系统页应直接复用这套模式
- 哪些内容属于页面个性，哪些内容属于平台通用规范

- 全局顶部结构

  - `DsRoleTag`
  - `DsStatusTag`

## 标准页面结构

后续后台标准列表页优先参考以下层级：

1. 全局顶部层
   - 品牌、通知、用户菜单、退出
2. 页面切换层
   - 轻量标签页
   - 关闭当前 / 关闭其他 / 关闭全部
3. 页面标题层
   - 面包屑
   - 页面标题
   - 页面说明
   - 页面级主操作 / 更多工具
4. 内容层
   - 统计卡片
   - 筛选查询
   - 数据列表
   - 分页

## 已确认的通用交互

### 页面头部

- 页面标题、面包屑、说明统一放在 `DsPageHeader`
- 页面主操作和更多工具统一使用 `DsHeaderActions`
- 面包屑属于页面定位信息，不和“关闭当前/其他/全部”混排

### 筛选查询

- 主筛选首屏保持一行优先
- 关键词、部门、状态、角色为高频筛选的标准组合
- 更多筛选作为展开面板，不占默认高度

### 列表头部

- 左侧：标题 + 说明
- 右侧：选择反馈 + 批量操作 + 工具按钮
- 批量操作未选中时保持禁用，不隐藏到布局跳动

### 表格

- 用户信息使用 `DsUserCell`
- 状态、角色、安全状态均使用统一标签/状态组件
- 行内操作外露“编辑 / 更多”
- 低频、危险、易误操作动作收进“更多”

### 分页

- 分页保持一行式
- 左侧总数，右侧页码、每页条数和跳转
- 不允许拆成上下两层

### 弹窗表单

- 按业务含义分组，不平铺成长表单
- 只读字段使用 `DsReadonlyField`
- 部门字段必须显示部门名称，不显示 id
- 角色字段使用扁平多选，不使用多级勾选

## 后续页面应直接复用的内容

以下页面优先复用用户管理页样板结构：

- `/system/department`
- `/system/role`
- `/system/log`
- `/system/config`
- `/system/dict`
- `/system/file`

复用时优先复用的是：

- 顶部层级结构
- 筛选条结构
- 列表头部结构
- 分页结构
- 弹窗表单结构

不要求完全照搬的是：

- 统计卡片数量
- 筛选字段组合
- 表格列内容
- 行内更多菜单项

## 明确不应复用为全局规范的部分

以下内容属于用户管理页业务个性，不应直接上升为所有页面统一规则：

- 用户字段集合
  - 手机号
  - 身份证号
  - 强制改密
  - 登录安全
- 用户账号安全相关弹窗文案
- 用户角色展示中的业务命名

## 当前样板价值

用户管理页现在的价值有三点：

- 它是基础平台 design-system 的第一张完整落地页
- 它验证了列表页、弹窗表单、顶部结构三类核心模式
- 它为后续系统页提供了可直接复用的实现基线，避免每页重新设计

## 后续维护规则

- 后续如果用户管理页继续做视觉或结构微调，必须同步检查是否影响样板地位
- 如果某处改动已经上升为平台通用规范，必须同步更新：
  - `docs/UI组件库规范.md`
  - `AGENTS.md`
- 如果某处只是用户管理页业务特性，不要误记为平台统一要求
