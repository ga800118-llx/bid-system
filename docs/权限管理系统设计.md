## 核心理念

把权限体系拆成两层：

业务系统（如招投标）
  注册自己的 Feature（菜单/按钮/字段权限）
               |
               v
     通用权限服务（可独立部署）
     Feature层：能不能操作
     DataScope层：能看哪些数据

用户 - 属于组织架构（树形部门） - 拥有多个角色
                                            |
                                Feature权限（能做哪些操作）
                                DataScope（能看哪些数据范围）
                                Field权限（能看到哪些字段）

## 二、权限类型

### 1. Feature 权限（功能操作）

权限码格式：模块:操作

示例：
- bid_project:read - 查看项目
- bid_project:write - 新增/编辑项目
- bid_project:delete - 删除项目
- bid_project:export - 导出
- finance:read - 财务模块读
- finance:pay - 付款操作

### 2. Field 权限（字段级）

示例：
- bid_project:field:ceilingPrice:read - 能看拦标价
- bid_project:field:bidBond:write - 能编辑保证金

### 3. Data Scope（数据可见范围）

每条规则是一个范围表达式，运行时由业务系统根据用户上下文计算。

| 规则 | 含义 | SQL 转化 |
|------|------|----------|
| OWN | 只看自己创建的 | WHERE user_id = :currentUserId |
| DEPT | 本部门 | WHERE dept_id = :userDeptId |
| DEPT_SUB | 本部门及下级部门 | WHERE dept_id IN (本部门+所有子部门) |
| DEPT_UP | 本部门及上级部门 | WHERE dept_id IN (本部门+所有父部门) |
| ALL | 全部 | 不过滤 |

---

## 三、权限运算规则

### 多角色 - 取并集

用户权限 = 角色A的Feature 并 角色B的Feature
用户数据范围 = 角色A的Scope 并 角色B的Scope（满足任一即可访问）
用户字段权限 = 角色A的Field 并 角色B的Field

### 实时生效

- JWT 只存 userId，不存权限
- 每次请求从 DB 加载权限
- 可加 Redis 缓存：变更权限时删缓存，下次请求重新加载

权限变更 - 发布前端刷新事件/清理缓存 - 下次请求重新加载 - 实时生效

## 四、数据库表结构

`sql
-- 角色表
org_role (
  id          BIGINT PRIMARY KEY,
  code        VARCHAR(50) UNIQUE COMMENT '角色编码',
  name        VARCHAR(100) COMMENT '角色名称',
  description VARCHAR(255),
  created_at  DATETIME
)

-- Feature权限（能做哪些操作）
org_role_feature (
  role_id      BIGINT,
  feature_code VARCHAR(100),
  PRIMARY KEY (role_id, feature_code)
)

-- 数据范围（能看哪些数据）
org_role_data_scope (
  role_id      BIGINT,
  module_code  VARCHAR(50) COMMENT '模块编码',
  scope_rule   VARCHAR(20) COMMENT 'OWN/DEPT/DEPT_SUB/DEPT_UP/ALL',
  PRIMARY KEY (role_id, module_code)
)

-- 字段权限（能看到哪些字段）
org_role_field (
  role_id     BIGINT,
  module_code VARCHAR(50),
  field_code  VARCHAR(50),
  can_read    TINYINT DEFAULT 1,
  can_write   TINYINT DEFAULT 0,
  PRIMARY KEY (role_id, module_code, field_code)
)

-- 用户-角色（多对多）
org_user_role (
  user_id BIGINT,
  role_id BIGINT,
  PRIMARY KEY (user_id, role_id)
)
`

## 五点一、当前实现

当前接口前缀使用项目现有规范：`/api/system`。

权限管理已实现接口：

```text
GET    /api/system/roles
POST   /api/system/roles
PUT    /api/system/roles/{id}
DELETE /api/system/roles/{id}
GET    /api/system/roles/permission-catalog
GET    /api/system/roles/{id}/permissions
PUT    /api/system/roles/{id}/features
PUT    /api/system/roles/{id}/data-scopes
PUT    /api/system/roles/{id}/fields
GET    /api/system/users/{id}/roles
PUT    /api/system/users/{id}/roles
```

权限检查相关实现：

- `@RequirePermission`：单权限点检查
- `@RequireAnyPermission`：多个权限点满足任一即可访问
- JWT 主要保存用户身份信息，权限由后端用户信息接口加载
- 前端菜单和按钮根据权限动态显示
- 用户管理、部门内用户列表已接入 DataScope
- 用户管理和操作日志已接入字段读写/展示过滤
- 权限拒绝返回统一 `403`，并写入操作日志 `AUTH / PERMISSION_DENIED`

已实现表：

- `org_role`
- `org_user_role`
- `org_role_feature`
- `org_role_data_scope`
- `org_role_field`

兼容约定：

- 当前用户主表仍为现有 `users`
- 新增 `org_user_role` 支持多角色
- 保存用户多角色时同步旧字段 `users.role`
- 含系统管理员角色时 `users.role = admin`，否则为 `user`
- 仍保留 `users.role` 和前端本地角色字段作为兼容信息，不再作为主要权限来源

当前权限点目录只覆盖基础平台：

| 模块 | 权限点 |
|------|--------|
| 部门管理 | 查看、新增、编辑、删除、设置负责人 |

| 操作日志 | 查看 |
| 系统配置 | 查看、新增、编辑、删除、启用禁用 |

## 六、权限检查流程

`
请求进入
    |
从 JWT 拿 userId
    |
从权限服务加载用户权限（当前查DB，后续可加缓存）
    |
  1. Feature 检查：@RequirePermission("bid:write")
     无权限 - 返回 403
     |
  2. DataScope 注入：自动拼接 SQL WHERE 条件
     查询时自动带上范围过滤
     |
  3. Field 检查：返回前端前过滤不可见字段
     Response 过滤掉 can_read=0 的字段
`

### Spring Boot 注解示例

`java
@RequirePermission("bid_project:write")
public Result saveProject(ProjectDTO dto) { ... }
`

`java
@DataScope(module = "bid_project")
public List<Project> list(ProjectQuery q) { ... }
`

## 八点一、新模块权限接入清单

适用范围：

- 后续新增业务模块，如项目管理、文件上传、报价、合同、审批等
- 目标是按统一套路接入 Feature、DataScope、Field 和操作日志

推荐按下面 7 步执行。

### 1. 先定义模块边界

先明确这个模块需要哪三类权限：

- Feature：能不能进入页面、查询、新增、编辑、删除、导出、提交审批等
- DataScope：能看自己的、本部门的、本部门及下级的，还是全部
- Field：哪些字段需要做可读/可写控制

判断原则：

- 列表、详情、按钮、弹窗动作，优先归到 Feature
- “能看哪些记录”，归到 DataScope
- “同一条记录里哪些字段可见、哪些字段可编辑”，归到 Field
- 如果对象更像“配置项”“字典项”“附件动作”，优先考虑对象级权限，不强行拆字段权限

### 2. 注册权限点目录

在 `RoleService` 里补模块目录：

- `featureCatalog()`：补功能权限点
- `dataScopeCatalog()`：如果该模块需要数据范围，补模块编码
- `fieldCatalog()`：如果该模块需要字段权限，补模块字段定义

约定：

- 模块编码统一，如 `bid_project`
- 权限码统一用 `模块:动作`，如 `bid_project:read`、`bid_project:update`
- 字段编码保持稳定，避免频繁改名导致历史权限失效

### 3. 后端接口接 Feature 权限

控制器入口统一加权限注解：

```java
@RequirePermission("bid_project:read")
@GetMapping("/api/project/list")
public ApiResponse list(...) { ... }
```

建议：

- 查询、详情、列表：`read`
- 新增：`create`
- 编辑：`update`
- 删除：`delete`
- 特殊动作单独建权限点，如 `export`、`approve`、`assign`

### 4. 查询接口接 DataScope

如果模块存在“谁能看到哪些数据”的问题，就在查询服务层接数据范围。

最低要求：

- 明确该模块按哪个字段过滤，比如 `dept_id`、`owner_id`、`created_by`
- 统一从 `RoleService.getDataScopeRuleForUser(username, moduleCode)` 取规则
- 在 SQL / QueryWrapper 层注入对应过滤条件

建议顺序：

- 先支持 `OWN / DEPT / ALL`
- 后续再扩 `DEPT_SUB / DEPT_UP`

### 5. 返回和写入接字段权限

如果模块确实需要字段级控制，按“读写分开”接入：

- 列表/详情返回前：按 `getReadableFieldsForUser` 过滤字段
- 新增/编辑保存前：按 `getWritableFieldsForUser` 或 `hasWritableField` 校验写入

推荐做法：

- 服务层统一组装 `readableFields`、`writableFields`
- 前端根据它控制列展示和表单禁用
- 后端继续兜底，不信任前端隐藏状态

注意：

- 像 `id`、`createdAt`、`updatedAt` 这类基础元数据，可继续作为固定返回字段
- 敏感业务字段再纳入字段权限控制，不必一开始追求全字段覆盖

### 6. 前端页面接菜单、按钮、列和表单

前端至少接四层：

- 路由和菜单：按 Feature 显示
- 按钮：按 Feature 显示
- 表格列/详情块：按 `readableFields` 显示
- 表单项/操作按钮：按 `writableFields` 控制可编辑

推荐模式：

- 页面初始化时拿当前模块的 `readableFields`、`writableFields`
- 列配置里加 `fieldCode`
- 统一用 `canReadField(fieldCode)`、`canWriteField(fieldCode)` 判断

### 7. 接操作日志和回归验证

模块接入完成后，至少补两类验证：

- 审计日志：
  - 成功操作要记录
  - 权限拒绝要记录
- 回归验证：
  - 管理员访问通过
  - 受限角色访问返回裁剪后的字段
  - 无写权限用户强行提交写请求返回 `403`
  - DataScope 外的数据不可见

建议最小验收清单：

- 角色页能配置该模块权限
- 非管理员能看到配置效果
- 刷新后权限回读一致
- 前端隐藏和后端拦截口径一致

### 当前经验总结

基础平台第一版已经验证过的接入模式：

- `system_user`：Feature + DataScope + Field
- `system_log`：Feature + Field
- `system_config`：Feature + 审计
- `system_dict`：Feature + 审计

因此后续新增模块时，优先按下面顺序判断：

1. 先接 Feature
2. 再判断是否需要 DataScope
3. 最后判断是否值得接字段权限

不是每个模块都必须同时接三层权限，按业务价值控制复杂度。

## 九、与现有招投标系统的集成

现有系统的用户/角色迁移：

1. 保留现有的 sys_user 表，通过 org_user 表关联或直接迁移
2. 现有 admin 角色 - 迁移为拥有所有 Feature + ALL DataScope 的系统管理员
3. 现有 user 角色 - 按具体功能分配对应权限码

Feature 码建议（招投标模块）：

| 权限码 | 含义 |
|--------|------|
| bid_project:read | 查看项目列表和详情 |
| bid_project:write | 上传文件/新增项目 |
| bid_project:delete | 删除项目 |
| bid_project:export | 导出项目数据 |
| bid_prompt:write | 修改 Prompt 配置 |
| bid_user:manage | 用户管理（admin 专属） |
