# 数据库设计说明

> 更新时间：2026-04-24
> 当前口径：以 `backend/local-schema.sql` 为准
> 适用范围：当前基础平台封板阶段 + 保留的历史业务表

这份文档用于说明当前项目数据库的真实结构、表分组、关键关联关系、初始化种子和后续开发约定。

## 一、当前数据库口径

### 本地开发

- 数据库类型：H2
- 本地库文件：`backend/.local-db/bid`
- 初始化脚本：`backend/local-schema.sql`

当前本地运行通过 Spring Boot 启动参数中的 `INIT=RUNSCRIPT` 执行 `local-schema.sql`。

### 生产环境

- 数据库类型：MySQL
- 当前推荐仍以 `backend/local-schema.sql` 作为结构基线参考

说明：

- `docs/开发文档/database_init.sql` 属于早期历史初始化脚本，不作为当前真实数据库口径
- 当前真正应该参考的是 `backend/local-schema.sql`

### 当前数据库设计原则

- `local-schema.sql` 只做建表、必要种子和兼容性补字段
- 不在初始化脚本里覆盖已有业务数据
- 新增基础平台表、字段、内置配置、内置字典时，优先走 `IF NOT EXISTS` 或兼容性 `ALTER`

## 二、表分组总览

当前数据库可以分成 6 组：

1. 用户与组织
2. 权限
3. 审计与系统能力
4. 字典
5. 历史业务表
6. 历史兼容表

### 2.1 用户与组织

| 表名 | 说明 | 当前状态 |
|---|---|---|
| `users` | 当前主用户表，登录、状态、部门、兼容角色字段都在这里 | 主用 |
| `org_department` | 部门树与负责人 | 主用 |

### 2.2 权限

| 表名 | 说明 | 当前状态 |
|---|---|---|
| `org_role` | 角色定义 | 主用 |
| `org_user_role` | 用户与角色多对多关系 | 主用 |
| `org_role_feature` | 功能权限 | 主用 |
| `org_role_data_scope` | 数据范围规则 | 主用 |
| `org_role_field` | 字段权限规则 | 主用 |

### 2.3 审计与系统能力

| 表名 | 说明 | 当前状态 |
|---|---|---|
| `system_operation_log` | 操作日志、权限拒绝日志、登录安全日志 | 主用 |
| `system_config` | 系统配置中心 | 主用 |
| `system_file` | 文件中心主表 | 主用 |
| `system_file_link` | 文件与业务对象关联表 | 主用 |
| `system_message` | 站内消息主表 | 主用 |
| `system_message_receiver` | 站内消息接收表 | 主用 |
| `system_todo` | 待办中心主表 | 主用 |

### 2.4 字典

| 表名 | 说明 | 当前状态 |
|---|---|---|
| `system_dict_type` | 字典类型 | 主用 |
| `system_dict_item` | 字典项 | 主用 |

### 2.5 历史业务表

| 表名 | 说明 | 当前状态 |
|---|---|---|
| `projects` | 早期项目管理与 AI 提取结果存储 | 保留，非当前基础平台重点 |
| `extract_logs` | 早期提取日志 | 保留 |
| `prompt_template` | 早期 Prompt 配置 | 保留 |

### 2.6 历史兼容表

| 表名 | 说明 | 当前状态 |
|---|---|---|
| `org_user` | 早期组织用户表，当前主链路已不再使用 | 保留兼容，不作为当前主口径 |

## 三、核心表说明

## 3.1 users

用途：

- 当前系统主用户表
- 登录认证、账号状态、部门归属、最近登录时间都从这里读取
- 当前仍保留 `role` 字段作为系统管理员兼容兜底

关键字段：

| 字段 | 说明 |
|---|---|
| `id` | 用户主键 |
| `username` | 登录账号，唯一 |
| `password` | 登录密码 |
| `role` | 兼容角色字段，当前仍保留 `admin/user` |
| `real_name` | 用户姓名 |
| `dept_id` | 所属部门，`NULL` 表示未分配 |
| `status` | 用户状态，`1=正常`、`0=禁用` |
| `failed_login_count` | 连续失败次数 |
| `last_failed_login_at` | 最近一次失败时间 |
| `last_login_at` | 最近登录时间 |

当前约定：

- `users` 是当前主用户表
- `org_user_role` 是当前主角色关系表
- `users.role` 仍保留用于兼容和系统管理员兜底

## 3.2 org_department

用途：

- 部门树结构
- 部门负责人
- 组织架构展示

关键字段：

| 字段 | 说明 |
|---|---|
| `id` | 部门主键 |
| `name` | 部门名称 |
| `parent_id` | 父部门 ID，根部门为空 |
| `manager_user_id` | 部门负责人 |
| `sort_order` | 排序值 |
| `status` | 状态，`1=正常`、`0=禁用` |

当前约定：

- 根部门当前固定种子 `id=1`
- 根部门名称默认是“我的公司”
- 前端展示时根部门名称优先从 `enterprise.name` 读取

## 3.3 org_role

用途：

- 角色定义

关键字段：

| 字段 | 说明 |
|---|---|
| `id` | 角色主键 |
| `role_code` | 角色编码 |
| `role_name` | 角色名称 |
| `description` | 角色说明 |
| `status` | 启用状态 |

当前内置角色：

- `admin` / 系统管理员
- `user` / 普通用户

## 3.4 org_user_role

用途：

- 用户与角色多对多关系

主键：

- `(user_id, role_id)`

当前约定：

- 用户新增、编辑、角色分配都以这张表为准
- 初始化时会根据 `users.role` 自动补主角色关系

## 3.5 org_role_feature

用途：

- 保存角色功能权限，例如 `system_user:read`

主键：

- `(role_id, feature_code)`

当前口径：

- 基础平台核心接口已接入真实权限校验

## 3.6 org_role_data_scope

用途：

- 保存角色在某模块下的数据范围规则

关键字段：

| 字段 | 说明 |
|---|---|
| `role_id` | 角色 ID |
| `module_code` | 模块编码 |
| `scope_rule` | 范围规则，如 `ALL`、`DEPT`、`OWN` |

当前已实际用于：

- 用户管理
- 部门内用户列表

## 3.7 org_role_field

用途：

- 保存角色在某模块下的字段可读/可写规则

关键字段：

| 字段 | 说明 |
|---|---|
| `role_id` | 角色 ID |
| `module_code` | 模块编码 |
| `field_code` | 字段编码 |
| `can_read` | 是否可读 |
| `can_write` | 是否可写 |

当前已实际用于：

- `system_user`
- `system_log`

## 3.8 system_operation_log

用途：

- 记录基础平台关键写操作
- 记录权限拒绝
- 记录登录成功、失败、锁定、退出

关键字段：

| 字段 | 说明 |
|---|---|
| `operator_username` | 操作账号 |
| `module` | 模块编码 |
| `action` | 动作编码 |
| `content` | 操作摘要 |
| `request_method` | 请求方法 |
| `request_path` | 请求路径 |
| `ip` | 来源 IP |
| `success` | 是否成功 |
| `error_message` | 错误信息 |

## 3.9 system_config

用途：

- 统一管理系统 key-value 配置

关键字段：

| 字段 | 说明 |
|---|---|
| `config_key` | 配置键，唯一 |
| `config_value` | 配置值 |
| `config_name` | 配置名称 |
| `description` | 配置说明 |
| `value_type` | `TEXT/NUMBER/BOOLEAN/JSON` |
| `group_code` | 配置分组 |
| `status` | 是否启用 |

当前已实际驱动：

- 企业名称
- 系统标题
- 密码策略
- 登录失败上限
- 会话超时
- 默认分页
- 侧边栏默认折叠
- 用户水印
- 文件中心存储配置
- 文件中心上传与预览规则配置
- 导出动作审计

## 3.10 system_file

用途：

- 统一存储文件中心上传文件元数据
- 与后续业务模块附件能力复用

关键字段：

| 字段 | 说明 |
|---|---|
| `storage_type` | 存储类型，第一版为 `LOCAL` |
| `object_key` | 文件对象键，文件中心内部唯一标识 |
| `original_name` | 原始文件名 |
| `extension` | 扩展名 |
| `content_type` | MIME 类型 |
| `file_size` | 文件大小，单位字节 |
| `file_hash` | 文件哈希，当前为 SHA-256 |
| `storage_path` | 本地实际存储路径 |
| `status` | 是否启用 |
| `uploader_id` | 上传人 |

说明：

- 第一版已支持列表、详情、上传、下载、启用/禁用
- 第一版只支持本地存储

## 3.11 system_file_link

用途：

- 管理文件和业务对象之间的关联关系

关键字段：

| 字段 | 说明 |
|---|---|
| `file_id` | 关联文件 |
| `biz_type` | 业务类型编码 |
| `biz_id` | 业务对象主键 |
| `category_code` | 附件分类编码 |
| `sort_order` | 排序 |
| `created_by` | 关联创建人 |

说明：

- 第一版后端接口已落地
- 前端业务关联管理页后续按业务模块接入

## 3.12 system_message

用途：

- 存储站内消息主内容

关键字段：

| 字段 | 说明 |
|---|---|
| `title` | 消息标题 |
| `content` | 消息正文 |
| `message_type` | 消息类型，如 `SYSTEM/NOTICE/TODO/WARNING` |
| `target_type` | 发送对象类型，当前支持 `ALL/USER` |
| `sender_id` | 发送人 |
| `sender_name` | 发送人名称 |

说明：

- 第一版支持后台发送站内消息
- 第一版不做模板、撤回、删除、外部通道

## 3.13 system_message_receiver

用途：

- 存储消息和接收用户之间的关系
- 记录已读状态和已读时间

关键字段：

| 字段 | 说明 |
|---|---|
| `message_id` | 消息主键 |
| `user_id` | 接收用户 |
| `read_status` | 已读状态，`0=未读`、`1=已读` |
| `read_at` | 已读时间 |

说明：

- 第一版支持未读数、单条已读、全部已读

## 3.14 system_todo

用途：

- 存储待办中心主数据
- 支撑个人待办列表、状态流转、导出
- 作为后续业务模块挂接待办能力的基础表

关键字段：

| 字段 | 说明 |
|---|---|
| `title` | 待办标题 |
| `content` | 待办内容 |
| `todo_type` | 待办类型，如 `GENERAL/SYSTEM/FOLLOW_UP` |
| `priority` | 优先级，如 `LOW/MEDIUM/HIGH/URGENT` |
| `status` | 状态，如 `PENDING/PROCESSING/DONE` |
| `biz_type` | 关联业务类型编码 |
| `biz_id` | 关联业务对象主键 |
| `assignee_id` | 处理人 |
| `assignee_name` | 处理人名称快照 |
| `creator_id` | 创建人 |
| `creator_name` | 创建人名称快照 |
| `due_at` | 截止时间 |
| `processed_at` | 完成时间 |

说明：

- 第一版一个待办只对应一个处理人
- 第一版创建待办时会自动发送一条 `TODO` 类型站内消息
- 第一版不做抄送、转派、催办、批量处理、审批流编排

## 3.15 system_dict_type / system_dict_item

用途：

- 统一管理字典类型和字典项

关系：

- `system_dict_type.id = system_dict_item.type_id`

当前已服务：

- 通用状态
- 成功/失败
- 用户角色类型
- 配置分组
- 配置值类型
- 消息类型
- 文件存储类型
- 标签颜色
- 操作日志模块
- 操作日志动作

## 3.15 projects

用途：

- 早期项目与 AI 提取结果存储

说明：

- 该表当前仍保留
- 但当前基础平台阶段不继续扩展这部分能力
- 后续如果业务模块重新启动项目管理，应再单独梳理这张表的口径

## 3.16 extract_logs / prompt_template / org_user

这些表当前都属于保留状态：

- `extract_logs`：早期提取日志
- `prompt_template`：早期 Prompt 配置
- `org_user`：早期组织用户表，当前主链路不再使用

它们不应作为当前基础平台数据库主口径。

## 四、关键关系

当前最重要的关系可以概括为：

```text
users.dept_id                -> org_department.id
org_department.manager_user_id -> users.id

org_user_role.user_id        -> users.id
org_user_role.role_id        -> org_role.id

org_role_feature.role_id     -> org_role.id
org_role_data_scope.role_id  -> org_role.id
org_role_field.role_id       -> org_role.id

system_dict_item.type_id     -> system_dict_type.id
```

当前权限主链路：

```text
users -> org_user_role -> org_role
                       -> org_role_feature
                       -> org_role_data_scope
                       -> org_role_field
```

## 六、当前数据库设计约束

### 6.1 当前主用户表是 `users`

不要把 `org_user` 误认为当前主用户表。当前系统真实登录、用户管理、权限加载都以 `users` 为准。

### 6.2 当前权限主来源是角色表体系

角色权限主来源：

- `org_user_role`
- `org_role_feature`
- `org_role_data_scope`
- `org_role_field`

但系统管理员仍保留 `users.role = admin` 的兼容兜底。

### 6.3 `local-schema.sql` 只做兼容初始化

后续新增基础平台字段或表结构时，应遵守：

- 不重建已有表
- 不清空已有数据
- 用 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`
- 用 `INSERT ... WHERE NOT EXISTS` 补种子

### 6.4 当前字段权限范围有意收住

当前字段权限只覆盖：

- `system_user`
- `system_log`

这是当前阶段的明确边界，不是漏做。

## 七、后续开发约定

新增模块时，数据库层建议按这个顺序判断：

1. 是否需要新业务表
2. 是否需要新系统配置项
3. 是否需要新字典类型或字典项
4. 是否需要注册新的 Feature / DataScope / Field 权限点
5. 是否需要写入操作日志

### 新增表建议

- 表名和字段命名延续当前风格
- 时间字段统一保留 `created_at`、`updated_at`
- 状态字段优先使用 `status`
- 可配置枚举优先考虑接入字典，而不是直接写死

### 新增基础平台种子建议

- 配置：写入 `system_config`
- 字典：写入 `system_dict_type` / `system_dict_item`
- 内置角色能力：必要时补 `org_role_feature` 等种子
