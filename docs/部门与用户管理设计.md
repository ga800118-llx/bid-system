## 一、数据库表结构

### 1.1 org_department / departments

当前本地实现使用部门实体 `Department`，H2 本地库由 `backend/local-schema.sql` 做兼容初始化。

核心字段：

```sql
id               BIGINT PRIMARY KEY
name             VARCHAR(100) NOT NULL
parent_id        BIGINT NULL
sort_order       INT DEFAULT 0
status           TINYINT DEFAULT 1
manager_user_id  BIGINT NULL
created_at       DATETIME
updated_at       DATETIME
```

约定：

- 根部门为“我的公司”，当前 id=1
- 其他部门挂到根部门下
- `manager_user_id = NULL` 表示未设置负责人
- 后续企业名称从系统配置读取

### 1.2 users

当前用户实体为 `User`，兼容基础平台字段。

核心字段：

```sql
id              BIGINT PRIMARY KEY
username        VARCHAR(50) NOT NULL UNIQUE
password        VARCHAR(255) NOT NULL
real_name       VARCHAR(50)
role            VARCHAR(50)
dept_id         BIGINT NULL
status          TINYINT DEFAULT 1
last_login_at   DATETIME
created_at      DATETIME
updated_at      DATETIME
```

约定：

- `dept_id = NULL` 表示未分配
- `status = 1` 正常，`status = 0` 禁用
- 当前阶段仍保留 `role = admin/user` 的临时角色字段
- 后续角色权限体系完成后，迁移到用户-角色关联表
- 当前角色权限骨架已新增 `org_user_role`，用户新增/编辑可写入多角色，同时兼容同步 `role`

## 四、部门管理

页面路径：`/system/department`

已实现能力：

- 左侧部门树，右侧部门内用户列表
- 部门搜索
- 新增、编辑、启用/禁用部门
- 部门负责人设置
- 负责人候选仅来自本部门正常用户
- 未分配用户节点
- 部门内用户搜索、分页
- 新增用户、编辑用户、批量启用/禁用、调部门、转未分配
- 删除部门前展示影响范围

### 4.1 当前接口

```text
GET    /api/system/departments/tree
POST   /api/system/departments
PUT    /api/system/departments/{id}
DELETE /api/system/departments/{id}
GET    /api/system/departments/{id}/delete-impact
GET    /api/system/departments/{id}/manager-candidates
GET    /api/system/departments/{id}/users
GET    /api/system/departments/unassigned/users
POST   /api/system/departments/{id}/users
POST   /api/system/departments/unassigned/users
PUT    /api/system/departments/users/{id}
PUT    /api/system/departments/users/status
PUT    /api/system/departments/users/dept
```

### 4.2 删除部门策略

当前统一采用“禁止删除”策略，不做级联删除。

删除规则：

1. 若存在子部门，禁止删除
2. 若存在用户，禁止删除
3. 删除前查询影响范围并提示
4. 只有没有子部门且没有用户时才允许删除

删除影响提示：

```text
确认删除部门
- 当前部门下还有 N 个子部门
- 当前部门及子部门下还有 M 个用户
- 请先调整子部门或用户后再删除
```

说明：早期文档中的“删除部门级联，用户转未分配”已废弃，以当前实现为准。

### 4.3 部门负责人规则

- 负责人必须是本部门用户
- 负责人必须是正常状态用户
- 用户调离原部门时，若该用户是原部门负责人，则清空原部门负责人
- 用户被禁用时，若该用户是任何部门负责人，则清空对应部门负责人
- MyBatis-Plus 更新 null 字段需要显式配置 `FieldStrategy.ALWAYS`

## 五、用户管理 V2

页面路径：`/system/user`

目标：从旧版“角色下拉 + 重置密码”升级为全局用户管理页。

### 5.1 页面布局

```text
┌────────────────────────────────────────────────────────────┐
│ 用户管理                                         新增用户   │
├────────────────────────────────────────────────────────────┤
│ 关键词：姓名/账号  部门：全部/具体部门/未分配  状态  角色   │
├────────────────────────────────────────────────────────────┤
│ 姓名 | 账号 | 所属部门 | 角色 | 状态 | 创建时间 | 操作       │
└────────────────────────────────────────────────────────────┘
```

### 5.2 筛选能力

- 关键词：匹配 `username` 或 `real_name`
- 部门：全部、具体部门、未分配
- 状态：全部、正常、禁用
- 角色：全部、管理员、普通用户
- 分页：默认 20 条

### 5.3 目标接口

```text
GET /api/system/users
  page
  size
  keyword
  deptId        具体部门 id；为空表示全部；特殊值 unassigned 表示未分配
  status        1=正常，0=禁用
  role          admin/user

POST /api/system/users
  username
  password
  realName
  role
  deptId
  status

PUT /api/system/users/{id}
  realName
  role
  status
  deptId

PUT /api/system/users/{id}/password
  password

PUT /api/system/users/{id}/status
  status

PUT /api/system/users/status
  userIds
  status

PUT /api/system/users/dept
  userIds
  deptId
```

### 5.4 新增/编辑规则

新增：

- 账号必填、唯一
- 初始密码必填，至少 6 位
- 姓名建议必填
- 所属部门可为空，表示未分配
- 状态默认正常

编辑：

- 账号不可编辑
- 可编辑姓名、角色、状态、所属部门
- 角色已支持多选，写入 `org_user_role`
- 重置密码使用独立操作

### 5.5 启用/禁用规则

- 禁用后该用户不能再次登录
- 当前 JWT 无状态，不强制踢出已有会话
- 禁用负责人时自动清空对应部门负责人

### 5.6 调部门规则

- 可调整到任意存在部门
- 可调整为未分配
- 若用户是原部门负责人，调离时自动清空原部门负责人
