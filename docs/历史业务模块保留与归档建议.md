# 历史业务模块保留与归档建议

> 更新时间：2026-04-21
> 目的：梳理当前仓库中仍保留的早期业务模块代码，明确哪些建议保留、哪些建议后续归档观察

## 当前背景

项目早期开发重点是：

- 项目上传
- PDF 文本提取
- AI 结构化抽取
- Prompt 配置

当前开发重点已经切换到基础平台：

- 部门管理
- 用户管理 V2
- 角色权限
- 操作日志
- 系统配置
- 字典管理

因此仓库里会同时存在两类代码：

1. 当前仍在持续维护的基础平台代码
2. 暂时不扩展、但仍保留的历史业务模块代码

这份清单的目的不是立刻删除，而是让后续接手的人知道哪些内容该怎么看。

## 二、建议后续归档观察的内容

这些内容不建议继续当成当前活跃开发模块理解，更适合后续做归档或二次清理。

### 备份文件

- `backend/src/main/java/com/bid/system/service/MiniMaxExtractService.java.bak`
- `frontend/src/views/PromptConfig.vue.bak`

说明：

- `.bak` 文件通常是人工备份
- 不属于应长期保留的标准源码形态
- 建议后续确认无参考价值后删除，或迁到单独 `archives/` 目录

### 早期样例 / 输出文件

- `backend/*.pdf`
- `backend/*.xlsx`
- `uploads/*.pdf`

说明：

- 更像调试输出、历史上传样例或本地运行产物
- 不建议继续混在主源码目录里

建议：

- 若仍要用于手工回归，可迁到单独样例目录
- 若只是遗留运行结果，可后续统一归档

### 早期路径兼容目录

- `C:/`

说明：

- 明显带有历史 Windows 路径迁移痕迹
- 当前是否仍被旧脚本引用，需要后续再确认

## 四、建议的后续清理顺序

如果后续要继续整理仓库，建议按下面顺序推进：

1. 先确认历史业务模块是否还会短期继续用
2. 再处理 `.bak` 文件
3. 再迁移历史 PDF / Excel 样例
4. 最后再决定是否拆分或下线早期项目上传相关页面和接口

## 五、当前建议

当前最稳的策略是：

- **不删除历史业务模块主代码**
- **先删除或归档明显的备份文件和运行产物**
- **通过文档明确“当前重点不是它们”**

这样既不会误伤后续可能还要用的业务代码，也能让新接手的人快速分清主次。
