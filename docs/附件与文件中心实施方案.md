# 附件与文件中心实施方案

> 创建时间：2026-04-24
> 当前定位：基础平台下一阶段优先能力
> 目标：沉淀一套可被后续业务模块复用的统一附件/文件能力

## 一、为什么先做这个

当前基础平台已经完成：

- 部门管理
- 用户管理
- 角色权限
- 操作日志
- 系统配置
- 字典管理

下一步如果继续完善基础平台，最值得优先补的是“附件与文件中心”。

原因：

1. 后续大多数业务模块都会依赖附件能力  
   项目、合同、审批、消息、待办等模块，基本都需要上传、下载、查看和关联文件。

2. 仓库里已有早期上传能力，但仍是业务写法  
   当前 `ProjectController / ProjectService / Admin.vue` 中的上传下载逻辑，适合作为历史实现参考，但还不是平台级能力。

3. 附件能力适合先做第一版闭环  
   它比定时任务、API Key、限流更贴近真实业务复用，也更适合作为当前基础平台的下一块通用底座。

## 二、这次要做成什么

这一阶段不做复杂对象存储平台，也不追求一次支持所有场景。

第一版目标是做出一套：

- 可上传
- 可下载
- 可查询
- 可做业务关联
- 可做权限控制
- 可做审计日志
- 后续可平滑扩展到本地 / OSS / S3

也就是说，这次做的是：

**“基础平台附件中心第一版”**

不是“完整文件存储平台终局版”。

## 三、第一版范围

### 3.1 本次要做

1. 文件中心数据库表
2. 文件上传接口
3. 文件下载接口
4. 文件列表查询接口
5. 文件详情接口
6. 文件逻辑删除/禁用能力
7. 文件与业务对象的关联能力
8. 文件中心管理页面
9. 基础权限点
10. 操作日志接入

### 3.2 本次不做

1. OSS / S3 真正接入
2. 图片/Office 在线预览
3. 分片上传
4. 断点续传
5. 秒传
6. 文件版本管理
7. 文件夹/目录树
8. 防盗链、签名 URL
9. 文件病毒扫描

这些都可以作为后续增强项，不放在第一版里。

## 四、建议命名

为了避免以后混乱，这一块建议统一叫：

- 中文：附件与文件中心
- 路由：`/system/file`
- 表前缀：`system_file_*`
- 模块编码：`system_file`

说明：

- “附件”强调业务关联
- “文件中心”强调平台复用
- 两个词一起用，既方便当前理解，也方便后续扩展

## 五、数据模型设计

第一版建议拆两张主表：

1. `system_file`
2. `system_file_link`

### 5.1 system_file

用途：

- 保存文件本体元数据
- 不直接依赖某个具体业务模块

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `storage_type` | 存储类型，第一版默认 `LOCAL` |
| `bucket` | 存储桶/空间名，第一版可为空 |
| `object_key` | 存储对象键，唯一 |
| `original_name` | 原始文件名 |
| `extension` | 文件扩展名 |
| `content_type` | MIME 类型 |
| `file_size` | 文件大小，字节 |
| `file_hash` | 文件哈希，第一版建议 SHA-256 |
| `storage_path` | 实际存储路径 |
| `status` | 状态，`1=正常`、`0=禁用/逻辑删除` |
| `uploader_id` | 上传人 |
| `created_at` | 创建时间 |
| `updated_at` | 更新时间 |

建议说明：

- `object_key` 作为文件中心内部唯一标识
- `storage_path` 用于本地存储实际落盘
- 后续切 OSS / S3 时，结构可以继续复用

### 5.2 system_file_link

用途：

- 保存文件和业务对象之间的关联关系

建议字段：

| 字段 | 说明 |
|---|---|
| `id` | 主键 |
| `file_id` | 文件 ID |
| `biz_type` | 业务类型，例如 `project`、`contract`、`notice` |
| `biz_id` | 业务对象 ID |
| `category_code` | 附件分类，例如 `main`、`contract_scan`、`notice_attachment` |
| `sort_order` | 排序 |
| `created_by` | 关联创建人 |
| `created_at` | 创建时间 |

建议说明：

- 以后任意业务模块都通过 `biz_type + biz_id` 关附件
- 这样不会把文件逻辑散落到每个业务表里

## 六、存储策略

### 6.1 第一版默认本地存储

建议新增系统配置：

- `file.storage.type = LOCAL`
- `file.local.base_path = uploads/files`
- `file.upload.max_size_mb = 100`

建议目录结构：

```text
uploads/files/
  2026/
    04/
      24/
        <object_key>.<ext>
```

这样做的好处：

- 不会把所有文件堆在一个目录
- 后续迁移对象存储时更容易对齐

### 6.2 文件命名

建议实际落盘文件名不要直接用原文件名，而是：

```text
<yyyyMMddHHmmss>_<随机串>.<ext>
```

或者直接用：

```text
<sha256前缀>.<ext>
```

原始文件名保存在数据库中，用于展示和下载时回传。

## 七、权限设计

第一版建议最少先有这些权限点：

| 权限码 | 含义 |
|---|---|
| `system_file:read` | 查看文件列表、详情 |
| `system_file:upload` | 上传文件 |
| `system_file:download` | 下载文件 |
| `system_file:delete` | 删除/禁用文件 |
| `system_file:link` | 维护文件与业务对象关联 |

当前不建议过早做字段权限。

这一块更适合：

- 功能权限
- 对象级权限
- 后续按业务再控制是否能看某对象下的附件

## 八、数据范围建议

第一版可以先不单独做文件中心自己的 DataScope。

原因：

- 文件本身通常依附业务对象
- 更合理的控制方式，是跟着业务对象走

建议当前策略：

1. 文件中心管理页先只有管理员或具备 `system_file:*` 的人可见
2. 后续业务模块读取附件时，先校验业务对象权限，再查 `system_file_link`

也就是说：

**文件中心第一版先不独立做 DataScope，而是复用业务对象权限边界。**

## 九、接口设计

建议接口前缀：

```text
/api/system/files
```

### 9.1 基础接口

```text
GET    /api/system/files
GET    /api/system/files/{id}
POST   /api/system/files/upload
GET    /api/system/files/{id}/download
PUT    /api/system/files/{id}/status
DELETE /api/system/files/{id}
```

### 9.2 业务关联接口

```text
GET    /api/system/files/links?bizType&bizId
POST   /api/system/files/links
DELETE /api/system/files/links/{id}
```

### 9.3 建议查询参数

文件列表：

```text
page
size
keyword        匹配原文件名
extension      按扩展名筛选
status         状态
uploaderId     上传人
dateFrom/dateTo
```

## 十、前端页面设计

### 10.1 路由

```text
/system/file
```

### 10.2 页面目标

做一个基础平台风格统一的“文件中心”页面：

- 统一 Layout
- `meta.title = 文件中心`
- 全宽内容区
- 标准表格
- 统一分页

### 10.3 页面结构建议

```text
文件中心                                         上传文件
--------------------------------------------------------
关键词   扩展名   状态   上传人   时间范围             重置
--------------------------------------------------------
序号 | 文件名 | 类型 | 大小 | 上传人 | 上传时间 | 状态 | 操作
--------------------------------------------------------
分页
```

### 10.4 操作建议

- 上传
- 下载
- 查看关联信息
- 禁用/删除

第一版不建议做复杂预览弹窗。

## 十一、操作日志接入

第一版建议记录这些动作：

| 模块 | 动作 | 说明 |
|---|---|---|
| `FILE` | `UPLOAD` | 上传文件 |
| `FILE` | `DOWNLOAD` | 下载文件 |
| `FILE` | `DELETE` | 删除文件 |
| `FILE` | `STATUS` | 启停文件 |
| `FILE` | `LINK` | 关联业务对象 |
| `FILE` | `UNLINK` | 解除业务对象关联 |

说明：

- 上传失败也应记录失败日志
- 下载是否全记录，第一版可以只记录后台管理下载

## 十二、字典和系统配置接入建议

### 12.1 字典

建议新增这些字典：

- `file_storage_type`
  - `LOCAL`
  - 后续可补 `OSS`、`S3`

- `file_link_category`
  - 第一版可以先不做太多内置项
  - 后续按业务需要再补

### 12.2 系统配置

建议新增：

- `file.storage.type`
- `file.local.base_path`
- `file.upload.max_size_mb`

后续可扩：

- `file.allowed_extensions`
- `file.allowed_content_types`

## 十三、和当前历史上传能力的关系

当前仓库已有这些早期能力：

- `ProjectController`
- `ProjectService`
- `Admin.vue`
- `projects.file_path / file_original_name / markdown_file_path`

这部分当前仍是：

**项目业务侧上传逻辑**

不是平台化文件中心。

建议处理方式：

1. 第一版文件中心先独立实现
2. 不急着重构旧项目上传
3. 等文件中心稳定后，再考虑把旧项目上传逐步迁移到新附件体系

这样风险最小，也不会把当前基础平台工作和历史业务逻辑绑死。

## 十四、实施顺序

建议分 5 步做：

### 第一步：表结构和配置种子

- 新增 `system_file`
- 新增 `system_file_link`
- 新增文件中心系统配置
- 新增必要字典

### 第二步：后端能力

- entity / mapper / service / controller
- 本地存储实现
- 上传 / 下载 / 列表 / 删除 / 关联接口
- 权限校验
- 操作日志

### 第三步：前端页面

- `/system/file`
- 上传按钮
- 列表和筛选
- 下载与禁用/删除

### 第四步：文档同步

- 数据库设计说明
- 基础平台帮助与使用手册
- 基础模块实施计划

### 第五步：回归验证

- 上传文件
- 下载文件
- 列表筛选
- 权限拦截
- 操作日志

## 十五、这轮做完后，基础平台会多出什么

如果这块做完，基础平台就不再只是：

- 用户
- 权限
- 配置
- 字典
- 日志

而会进一步拥有：

- **统一附件能力**

这会直接提升后续业务模块开发效率。

后面再做：

- 站内消息
- 导入导出
- 个人中心
- 待办中心

整套基础平台就会更完整。

## 十六、建议结论

当前建议直接按这个口径推进：

1. 先做“附件与文件中心第一版”
2. 先支持本地存储
3. 先做基础权限和日志
4. 暂不重构旧项目上传
5. 做完后再考虑站内消息或导入导出

这是当前阶段最稳、最有复用价值的一步。
